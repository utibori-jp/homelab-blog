## Homelabのはじまりは唐突に

Homelabの始まりは、結構しょうもないものでした。

安くなっていたという理由で衝動買いしたMiniPCを、ちょっと遊んでしばらく放置してたら、パスワードを忘れてしまい初期化するしかなくなってしまいました。そこで、どうせ初期化するならもうOSごと入れ替えようと思い立ったのが、事の始まりでした。

この記事では、「具体的にどんなコマンドを叩いたか」という手順の話はしません。僕が普段遊んでるHomelabがどんな感じなのか、そのあたりを書けたらと思っています。
Homelabって興味はあるけど意外と何がいいのかわからない、といった人の参考になれば幸いです。

まずは、うちのHomelabの全体像をざっくり眺めてみてください。

## まずは全体像から

MiniPC上ではProxmoxがOSとして動いていて、その上にUbuntu ServerのVMが3台乗っかっています。このVM3台で、K3sクラスタを構成しています。

図にすると入れ子が多くて少しごちゃっとしていますが、外側から順に見ていくとシンプルです。いちばん外が家のネットワーク(192.168.0.0/24)で、そこにノートPCとMiniPCがぶら下がっています。MiniPCの中ではProxmoxが動いていて、さらにその内側に、Proxmox SDNで切った隔離ネットワーク(10.10.0.0/24)があります。K3sの3ノードはこの隔離ネットの中にいて、外とはSDNのgateway(10.10.0.1)経由でやりとりする、という構造です。

個人的なこだわりポイントは2つあります。
1. VMはTemplate + TerraformでIaC化する
2. VMはProxmox上のプライベートサブネット(10.10.0.0/24)に配置する

一つずつ話していきます。

### VMはTemplate + TerraformでIaC化する

始めProxmoxを入れて遊んでいた時は、[Proxmox VE Scripts](https://community-scripts.org/scripts)を使ってみたり、手動でぽちぽちサーバを増やしていました。
何となく動かしてみる分には全くそれで問題なかったのですが、後述するK3sクラスタを組もうってなったタイミングで、まったく同じ作業を何回もやることになるのはしんどいなと。また、今回MiniPCで動かしている以上、このMiniPcが逝ったら全て終わる状況なわけで、もしそうなったら、また一から手順を調べてやるのは流石にめんどくさい。
というわけでできるだけ自動化したいよねってことでIaC化してみることにしました。

ちなみに、僕が採用した方法はちょっと変わっています。Proxmoxの機能でVMテンプレートを作成し、そのテンプレートをTerraformで任意の数だけ立てる、という方法です。

教科書的にきれいなのは、TerraformでプレーンなVMを立てて、Ansibleでノードを設定していくやり方だと思います。ただ僕の場合、初期設定をのぞくとAnsibleの出番がほとんど無さそうでした。そのためだけに新しく環境を用意するのは、~~めんどくさい~~リターンが大きくないなと判断して、今回はAnsibleを見送ってこの方法に落ち着きました。（……まあ、結局なんやかんやあって後からAnsibleを入れたんですけどね。）

なお、VM自体は他の用途でも使うかもしれないので、K3s周りのソフトウェアは意図的にテンプレートへ入れていません。こっちは手動で入れました。

VMテンプレートはドライブに保存して誰でも見れるように公開してるので、このVMをダウンロードして`terraform apply`すれば、いくつでもだれでも好きなだけUbuntu Serverを立ち上げられます。興味ある方はぜひ試してみてください。

### VMはProxmox上のプライベートサブネット(10.10.0.0/24)に配置する

手動ポチポチ時代は、各サーバを直接家のルーターと同じネットワークに参加させていました。これをK3s導入のタイミングで、Proxmox上のプライベートサブネットにまとめて引っ越しました。あわせて、個々のサービスをLXCやVMとして建てるのをやめ、全てK3s上のアプリケーションとして動かす方針に変えています。この「ネットワークを分ける」と「アプリをK3sに寄せる」は、僕の中ではセットの判断でした。

ネットワークを分けたのは、家のルータから見た構成をすっきりさせたかったからです。homelabの通信は全てProxmoxの内側で完結するので、家のネットワークにhomelab由来のトラフィックが混ざりません。IPアドレスも、家のルータのDHCP任せではなく自分の管理下に置けます（VM自体のIPはTerraformで固定、各アプリのIPはK3s内で払い出し）。

アプリをK3sに寄せたのは、IaC化のしやすさが大きいです。LXCやVMで個別に建てていた頃と違って、各アプリをマニフェストとして宣言的に管理できるようになりました。

ただ、ネットワークを分けたことで、今度は手元のノートPCからノードへ直接アクセスしづらくなる、という副作用も出てきました。このあたりにどう対処したかは、後半の「詰まったところ」で触れます。


## なぜVM3台のK3sにしたか

なぜわざわざK3sクラスタなんてものを組んでいるのか、その動機についても少し触れておきます。

そもそもの発端は、Proxmoxで遊んでいるうちに、だんだん「この上でKubernetesを動かしてみたい」という欲が出てきたことでした。とはいえ当時の僕のKubernetesの理解は、複数のコンテナをまとめて管理してくれるらしい、くらいのもの。実際どうやってコンテナを管理していくのかは全く知らなかったですし、もちろん自分でクラスタを組んだこともありませんでした。

ただ、Kubernetesを触るだけなら、MinikubeやKindでシングルノードを立てれば済む話でもあります。それでも僕がVMを3台立てる構成を選んだ理由はシンプルで、せっかくProxmoxがあるんだから、ノード同士がネットワーク越しに通信する「ちゃんとした」マルチノード構成を組んでみたかったからです。台数はmaster×1 + worker×2にしました。複数ノードに処理が分散する様子を眺められて、かつMiniPC1台のリソースにも収まる、僕にとっての最小構成がこれでした。

ディストリビューションにK8sそのものではなくK3sを選んだのも、同じ発想です。MiniPC1台に全部を詰め込む以上、リソースは軽いに越したことはありません。それに、はじめから全機能を使いこなせるわけでもないので、まずは軽量なK3sで十分だと判断しました。

では、この構成を実際どう組んでいるのか、もう少しだけ中身を覗いてみます。

## 構成を少し細かく

「なぜこの構成か」はここまでで触れたので、細かいところは早見表でまとめておきます。どんな部品でできているのか、興味のある方向けに。

| 項目 | 値 |
|------|-----|
| ノード | 3台（home-lab-1〜3 / master×1, worker×2） |
| CPU / MEM / Disk | 4 core / 6GB / 64GB（各ノード） |
| ノードIP | 10.10.0.11 〜 .13 |
| ネットワーク | Proxmox SDN で 10.10.0.0/24 を分離（gateway 10.10.0.1 / SNAT有効） |
| VM作成 | Terraform（Provider: bpg/proxmox）+ cloud-init でテンプレートからclone |
| 詳細 | homelab-infra リポジトリ参照 |

コードの中身まで気になる方は、homelab-infra リポジトリを覗いてみてください。ここまでサラッと書きましたが、実際に組み上げるまでには、いくつか詰まったポイントもありました。

## 詰まったところ

ここまでサラッと書いてきましたが、もちろんすんなり組めたわけではありません。特に印象に残っている詰まりどころを2つ挙げておきます。

1つ目は、Terraformのプロバイダ問題です。最初はAIに勧められるまま、昔から定番の `telmate/proxmox` というProviderでコードを書いていました。ところが、当時の僕の環境（Proxmox 9.x）ではどうにもうまく動かせず、しばらくハマっていました。正直なところ、これがProvider側の対応状況によるものだったのか、自分の書き方の問題だったのかは、いまでもはっきりとは切り分けられていません。

ただ、調べていくうちに、より活発にメンテナンスされている `bpg/proxmox` というProviderの存在を知り、そちらに乗り換えました。AIが出してくるコードはどうしても `telmate` 前提に引っ張られがちだったので、最終的には公式のサンプルやドキュメントを横に置きながら、自分で書き直しています。今のAIに同じことを聞けば、最初から `bpg` 版でスッと出してくれるのかもしれませんね。

2つ目は、ノードにSSHが繋がらない問題です。これには2段階のつまずきがありました。

まずは構成上の話。Proxmoxは家のネットワークにいる一方で、K3sのノードは隔離ネットワークの中にいるので、手元のノートPCからは直接手が届きません。最初はProxmoxを踏み台にした多段SSHでしのいでいましたが、その後Tailscaleを入れて各ノードを同じtailnetに放り込んだことで、踏み台なしに直接SSHできるようになりました。

ところが、Tailscaleを入れたのに、なぜか外出先からだけ繋がらない、という現象に悩まされました。原因は拍子抜けするほど単純で、`.ssh/config` に書いていたIPアドレスの設定ミスです。Tailscaleを入れる前、ローカルで開発していた頃の名残で、家のルータのDHCPが払い出したIPをそのまま設定に残していたんです。このIPは家の中なら届きますが、外に出ると使えません。ここをTailscale側のIPに書き換えて、ようやく場所を問わず繋がるようになりました。

## さいごに

唐突に始まったHomelabでしたが、振り返ってみると学びの多い遊びでした。普段あまり意識しない家のネットワークと正面から向き合えたのは、いい機会だったなと思います。

個人的に感動したポイントを2つだけ。1つは、`terraform apply` を一発打つだけでVMが3台まとめて立ち上がった瞬間です。手動でポチポチVMを作ってウンザリした後だっただけに、その快適さは格別でした。コードになっていればAIにも相談しやすいですし、コメントで「なぜこうしたか」を残しておけるのも気持ちがいい。もう1つは、ノードにSSHが通った瞬間。社会人最初のプロジェクトで初めてSSHしたとき以来の、あの「繋がった！」という小さな感動が蘇りました。

この記事ではProxmoxとVM、ネットワークまわりが中心でしたが、肝心の「この上で何を動かしているのか」――ArgoCDによるGitOpsや、先ほど顔を出したTailscaleの設定まわり――については、また別の記事で書いていく予定です。よければ次回もお付き合いください。
